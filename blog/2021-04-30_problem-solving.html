A simple problem that got me stumped

This case is from FizzBuzz exercise in Sprint 4. I was writing multiple conditions by using if, and I was pretty confident with the conditions I set, but I still got an error.

Problem

I kept receiving an error while I was trying to build a function to narrow down which number should return certain string. 

Error: if the number is a multiple of three AND five, fizzbuzz should return the string 'FizzBuzz'
Techniques used
  • Google — Resources I was looking for were mainly to confirm if the grammar of the code was correct. I also searched for alternative solutions without using the initial approach.
  • Console.logging — I frequently used this for checking returned values.
  • Pseudocode — After using this technique, I found out why my initial attempt was incorrect. It helped me discover what to think, and in what order..
My journey

So, this was my initial attempt. I wrote this while I was reading the instructions. Needless to say, I was just following the instructions as written without thinking about the context in which the code would be executed. This will return "Fizz" when you run a console.log check with multiples of 15. 

function fizzbuzz(num) {
  if (num % 3 === 0) {
    return "FizzBuzz";
  } else if (num % 5 === 0) {
    return "Fizz";
  } else if (num % 15 === 0) {
    return "Buzz";
  } else {
    return num;
  }
}

The grammar was correct. So was the spelling. I nearly ran out of options to try out. Then, I remembered the technique of pseudocode. I wrote down what this code is supposed to do in human language as well as its returned value just in case. Then look what I found! 

// If the number is multiples of 3, it will be 3, 6, 9, 12, 15, 18, 21, 24, 27, 30... 
// If the number is multiples of 5, it will be 5, 10, 15... WAIT??

I realised something went wrong in my writing when I was picking up multiples of 5. Yes, I found a conflict here, and spotted why it returned "Fizz" whenever I tested with multiples of 15. So I moved this line to the top like the code below. 

// If the number is multiples of 15, it will be 15, 30, 45...
// If the number is multiples of 3, it will be 3, 6, 9, 12, 18, 21, 24, 27...
// If the number is multiples of 5, it will be 5, 10, 20, 25...

And the final outcome is this. 

function fizzbuzz(num) {
  if (num % 15 === 0) {
    return "FizzBuzz";
  } else if (num % 3 === 0) {
    return "Fizz";
  } else if (num % 5 === 0) {
    return "Buzz";
  } else {
    return num;
  }
}
Takeaways

Thinking everything only in my head is one way ticket to the world of confusion. When I encounter any tech issues (big or small), I tend to estimate scope as too broad fro me to handle on my own. My initial action most of the time was to search the internet to seek resources so that I can solidify my understanding on what I'm trying to do. Then I dig deeper and deeper... The worst part is I enjoy it when I have extra time. When I don't, I feel rushed and just end up with a big handful resources with countless tabs opened on my browser.

How might I do it differently? I still think that looking for resources on the internet is not so bad considering I'm still learning the basics. Within flipped learning, this is probably the way to go. However, I really must time it! I definitely need to know how long to be given for the research even though I have a thirst for new information. I'm also keen to experiment with pair work if people are available. Problems like this could be spotted by other people easily, and I could save tons of time over all.

Successful problem solving

It took me quite a while to understand how to manipulate complex objects, and I'm still working on it. As much as I love (or make efforts) to simplify things, I somehow produced complicated things in the initial attempt. This is a good example of how I could turn something complicated into simple during an exercise, Gradebook in Sprint 4.

Problem

I had a lucky guess that passed compiling a code which I didn't fully understand why it worked.

Techniques used
  • Asking coaches for help — During weekly catch-up with my facilitator, I explained my uncertainty about the code I wrote.
My journey

Let's get straight to the point. This story is not about how I amazingly solved my problem on my own. This is about how having another eye will help me learn more and better.

I haven't got a solid understanding on manipulating complex objects yet, but I'm getting there(...growth mindset). So I took a chance in the weekly catch-up with my facilitator and shared the code that I was working on. After I got my answer, my facilitator provided me with awesome feedback on another part of the code which I never thought would recieve feedback.

What I'm trying here is to assign the keys and values to the object I created earlier. 

for (let i = 0; i < students.length; i++) {
  const x = {
    testScores: scores[i]
  }; 
  Object.defineProperty(gradebook, students[i], {
    value: x,
    writable: true,
    enumerable: true,
    configurable: true
  });
}

And this is a suggestion I received. (Wow, so clean!) Absolutely, pennies from heaven. This was my first "aha!" moment. 

for (let i = 0; i < students.length; i++) {
  gradebook[students[i]] = { testScores : scores[i] };
}
Takeaways

I just witnessed the power of eyes from others. I realised there are so many things that are happening right in front of me, but I hardly notice the opportunities to improve my skills just because I'm too focused on the problem I'm currently solving. Now I totally get why so many people recommend to do pair work.

Problem solving techniques and process

This is my top 5 problem solving techniques and process, and how they have treated me during my learning journey so far.

  1. Googling — Searching for the right resources to solve my problem was hard because I can easily get overwhelmed by tons of information. Without timing the length of research, this technique can interfere with my primary goal. The key to use this more successfully is to manage my schedule.
  2. Console.logging — This is my new habit now, and I love doing it because I can verify what I want to do in code.
  3. Pseudocode — Surprisingly, this is my dark horse. The process of converting from human language to computer language teaches me how I need to think in code. I normally write a code per one action, which sometimes makes the code complex. But this is definitely a good starting point.
  4. Trying something — Definitely trying something whether I know it well or not, will return something to work on next.
  5. Reading error messages — I found this a bit tricky because error doesn't always help me fix codes. The relevance is not always true. Sometimes the line the error occurs turns out correct, and the whole problem was just a spelling mistake. Not always reliable technique.

And these are the techniques and processes I've used least. Here are some comments about why I used them less or didn't use them at all until now.

  • Rubber ducky method — To be honest, I haven't tried this technique because I try other techniques that I think help me find a solution quickly. And writing pseudocode while talking could probably be similar.
  • Asking your peers for help — The main reason why I haven't actively looked for this opportunity is simply we are remote, and setting up pair work may not be suitable during the foundation period. Now that I know the power of eyes from others, I'm definitely keen in the future!
  • Asking coaches for help — As long as availability is open, I'm up for it.
  • Improving your process with reflection — I guess writing blog articles falls this category, perhaps? Revisiting things done and documenting them in a way that others can understand is absolutely powerful.

Lastly, one thing I found interesting through this exercise of problem solving is the importance of planning. Apart from actual project planing, I used to call this phase wire-framing in my previous career as a designer. Wireframe is the very beginning of a design process, and it can be done as hand-drawn sketch. This is basically a blueprint of the product.

I realised I went through quite similar process with pseudocode. Although it's in written format, pseudocode defines and clarifies the features of my idea. I'm on the track with improving this skill, but I believe writing pseudocode will save heaps of my time and enhance productivity.
